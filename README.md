# FitMe360 — Android (Kotlin + MediaPipe)

بازنویسی نسخه‌ی پایتونی روی اندروید با تمرکز روی دقت بیشتر و اجرای native.

## چرا این معماری؟
- کد قبلی (`Tkinter` + `cv2.imshow`) اصلاً روی اندروید اجرا نمی‌شد.
- به‌جای `cvzone.PoseModule` از **MediaPipe Tasks Vision – Pose Landmarker** استفاده شده: مدل رسمی، on-device، سریع‌تر و دقیق‌تر.
- به‌جای چسباندن ساده‌ی تصویر لباس روی یک نقطه (شونه)، از **warp چهار نقطه‌ای** (دو شونه + دو باسن) با `Matrix.setPolyToPoly` استفاده شده، پس لباس با چرخش و خم شدن بدن هم‌راستا می‌مونه.

## مراحل راه‌اندازی

1. **مدل Pose Landmarker را دانلود کنید:**
   از https://ai.google.dev/edge/mediapipe/solutions/vision/pose_landmarker
   فایل `pose_landmarker_full.task` (یا نسخه‌ی `lite` برای گوشی‌های ضعیف‌تر) را در مسیر زیر بگذارید:
   ```
   app/src/main/assets/pose_landmarker_full.task
   ```

2. پروژه را در Android Studio باز کنید (Gradle sync خودش وابستگی‌ها را می‌گیره).

3. روی یک دستگاه واقعی اجرا کنید (شبیه‌ساز برای دوربین/GPU delegate مناسب نیست).

## فایل‌ها
| فایل | نقش |
|---|---|
| `PoseLandmarkerHelper.kt` | اجرای مدل Pose در حالت LIVE_STREAM |
| `ClothOverlayView.kt` | ترسیم فریم دوربین + warp لباس روی بدن |
| `MainActivity.kt` | CameraX، انتخاب لباس از گالری، اتصال قطعات به هم |
| `activity_main.xml` | رابط کاربری ساده (دکمه‌ی انتخاب لباس + دکمه‌ی Debug) |

## به‌روزرسانی: انتخاب لباس از گالری گوشی + حالت دیباگ

- دیگه نیازی به گذاشتن PNG لباس در `assets/shirts/` نیست. دکمه‌ی **«انتخاب لباس از گالری»** گالری گوشی رو باز می‌کنه و همون عکسی که کاربر انتخاب کنه مستقیم بارگذاری می‌شه.
- فرض شده عکس‌های لباس از قبل crop شدن و حاشیه‌ی اضافه ندارن (طبق چیزی که گفتید)، پس دیگه trim خودکار انجام نمی‌شه.
- دکمه‌ی **«Debug: ON/OFF»** یک لایه‌ی تشخیصی روی تصویر می‌کشه:
  - **نقطه‌های قرمز**: همه‌ی ۳۳ لندمارک بدنی که مدل تشخیص داده. اگه این نقطه‌ها اصلاً روی بدن نمی‌شینن یا اصلاً دیده نمی‌شن → مشکل از خود مدل/تشخیص pose هست، نه از منطق fit لباس.
  - **نقطه‌های سبز بزرگ‌تر**: دقیقاً همون ۴ نقطه‌ای که برای fit لباس استفاده می‌شن (دو شونه + دو باسن).
  - **خط زرد**: چهارضلعی‌ای که لباس قراره داخلش warp بشه — اگه لباس با این چهارضلعی فرق داره ولی خود چهارضلعی درست روی بدن نشسته، مشکل از نسبت‌های padding توی `ClothOverlayView.kt`. اگه خود چهارضلعی هم غلطه (خیلی کوچیک/جابه‌جا)، مشکل از mapping مختصات یا rotation دوربینه.
  - **متن‌های بالای صفحه**: تعداد لندمارک شناسایی‌شده، مختصات نرمالایز شونه/باسن، اندازه‌ی فریم دوربین در برابر اندازه‌ی View، و اندازه‌ی تصویر لباس بارگذاری‌شده.

با این اطلاعات یه اسکرین‌شات (Debug: ON) بفرستید تا دقیقاً بگم مشکل کجاست: خود تشخیص pose، mapping مختصات، یا فقط تنظیم padding.

یادآوری: مشکل تأخیر فریم/نتیجه (lag) هم قبلاً حل شده — هر فریم با timestamp خودش کش می‌شه و نتیجه‌ی pose دقیقاً با همون فریمی که روش محاسبه شده نمایش داده می‌شه، نه با جدیدترین فریم موجود.

اگه اندازه یا موقعیت لباس هنوز درست نیست ولی نقطه‌های دیباگ (قرمز/سبز/زرد) درست روی بدن می‌شینن، فقط کافیه این سه مقدار توی `ClothOverlayView.kt` رو تنظیم کنید:

| متغیر | تأثیر |
|---|---|
| `shoulderPaddingRatio` | لباس چقدر از هر شونه بیرون بزنه (بزرگ‌تر = لباس گشادتر) |
| `topPaddingRatio` | یقه چقدر بالاتر از خط شونه کشیده بشه |
| `bottomOvershootRatio` | لباس چقدر پایین‌تر از خط باسن بره |

## نکات بهبود دقت بیشتر (قدم بعدی، اختیاری)


- **Segmentation**: اضافه کردن `Selfie Segmentation` تا لباس درست پشت دست/مو قرار بگیره (الان لباس همیشه روی همه‌چیز کشیده می‌شه).
- **Smoothing**: یک فیلتر ساده (مثلاً میانگین متحرک روی مختصات لندمارک‌ها در ۳-۵ فریم آخر) بگذارید تا لرزش لباس کم بشه؛ دقیقاً همون کاری که در پروژه‌ی BallTracker با فیلتر Kalman انجام داده‌اید، این‌جا هم برای ثبات لندمارک‌ها مفیده.
- **YUV→RGB سریع‌تر**: تبدیل فعلی از JPEG واسط استفاده می‌کنه (ساده ولی کندتر)؛ برای فریم‌ریت بالاتر یک مبدل RenderScript/OpenGL جایگزین کنید.
- **دقت لباس با چرخش (yaw) شدید**: چون فقط ۴ نقطه‌ی صلب استفاده می‌شه، در چرخش زیاد (پروفایل کامل) لباس واقع‌گرا به‌نظر نمی‌رسه؛ برای اون سطح باید به مدل‌های AR واقعی مبتنی بر mesh بدن (مثل MediaPipe Holistic + مش سه‌بعدی پارچه) رفت که پیچیدگی بسیار بیشتری داره.

⚠️ این کد در Android Studio تست کامپایل نشده (این محیط SDK اندروید ندارد) — منطق و API-هاش درست‌اند ولی قبل از build نهایی یک بار Gradle sync/run لازمه تا مشکلات جزئی (نسخه‌ی وابستگی‌ها و مانند آن) دیده بشه.

## گرفتن APK بدون نصب چیزی (GitHub Actions)

این پروژه یک workflow آماده در `.github/workflows/build.yml` داره که به‌صورت خودکار APK می‌سازه:

1. یک ریپوی جدید (خصوصی یا عمومی) روی GitHub بسازید.
2. کل محتوای این پوشه رو push کنید:
   ```bash
   git init
   git add .
   git commit -m "FitMe360 Android"
   git branch -M main
   git remote add origin https://github.com/USERNAME/REPO.git
   git push -u origin main
   ```
3. مدل `pose_landmarker_full.task` رو قبل از push در `app/src/main/assets/` بذارید (طبق بخش «مراحل راه‌اندازی» بالا) — وگرنه اپ می‌سازه ولی موقع اجرا کرش می‌کنه چون مدل پیدا نمی‌شه. (تصاویر لباس دیگه نیازی به قرار گرفتن در assets ندارن؛ از گالری گوشی انتخاب می‌شن.)
4. برید به تب **Actions** توی ریپوی خودتون؛ workflow به اسم "Build APK" خودش بعد از push اجرا می‌شه (چند دقیقه طول می‌کشه).
5. وقتی سبز شد، روی همون run کلیک کنید؛ پایین صفحه بخش **Artifacts** یک فایل zip به اسم `FitMe360-debug-apk` هست — دانلودش کنید، داخلش `app-debug.apk` هست، مستقیم روی گوشی نصب می‌شه (باید "نصب از منابع ناشناس" رو در گوشی فعال کنید).

اگه بعداً بخواید بدون push دستی هم اجراش کنید، از تب Actions → workflow "Build APK" → دکمه‌ی **Run workflow** استفاده کنید (به لطف `workflow_dispatch` که در فایل تعریف شده).
